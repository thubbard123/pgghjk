For example, if you are producing print-based output, you may view your output and find it necessary to apply page breaks to some styles to make sure they start on a new page (see Setting Page and Column Breaks). After building the output again, you may notice that your generated table of contents (TOC) does not look exactly the way you want, so you make some design changes to it (see Using Styles to Determine the Look of a Print TOC). Each time you build, review, and refine your target, it moves closer and closer to its final state, until it is finally "ready."

 features to test

In addition to proofreading every section of the output, you need to test and review all of the features that you have included in your project, such as:

  • Autonumbers
  • Browse sequences
  • Conditional text
  • Context-sensitive Help (CSH)
  • Footnotes
  • Glossary
  • Headers and footers
  • Images
  • Index
  • Multimedia
  • Navigation links (e.g., cross-references)
  • Skin
  • Slideshows
  • Table of contents
  • Tables
  • Text boxes
 tips

Some features (e.g., TOC, index, cross-references, browse sequences, Help control links) require you to build and view the output for the entire target before testing them. However, you can test other features (e.g., images, conditional text) as you develop and preview individual topics.

One tool that can be extremely useful when it comes time to test and review your output is MadCap Analyzer. You can use 's built-in Analyzer or the more full-featured external Analyzer. The tool can be used to find and fix broken links, undefined variables, topics not indexed, and much more. See About Analyzer.

It is also a good idea to keep a checklist of all the features you need to test throughout your project. Each time you add a new feature to your project, add it to your checklist as well.
