Best Practices

Following are some of the best practices that may make your life easier when using .

global project linking—use with condition tags

There are many ways to single-source content in , and at many different levels. One of the most powerful ways to take advantage of single-sourcing is to do so at the broadest level—across multiple projects. It's called "Global Project Linking."

If you have multiple projects, you may find that you use the same content or files in some of those projects. Maybe there are a number of identical topics that you add to each project. Perhaps you have created a stylesheet that you copy and reuse in each project. Rather than creating this information separately, or relying on copy-and-paste actions, you can save time and ensure consistency by maintaining that information in just one project. When you want to use a file in another project, you simply import it by using a project import file. This creates a link between the source project file and the imported file in the secondary project. In the future, when you need to make changes to the file, you do not need to do so in multiple places, but rather in just one place—in the source project.

If you decide to take advantage of Global Project Linking, it is highly recommended that you use condition tags when doing so. By applying condition tags to your files (e.g., topics, images, stylesheets, page layouts), you can more easily control which files are imported into (or excluded from) a particular project. This is especially true if you elect to use the reimport feature, which automatically imports files that you specify when you generate targets.

See Global Project Linking—Importing Files from Other Projects and About Condition Tags.

 index—create using index entry mode and saved layouts

Creating an index is an extremely important part of developing a project. It is a task that is part science-part art, and it often involves virtually every topic in your project. Therefore, it can be one of the most time-intensive tasks in developing a project. There are ways, however, to make the process a little easier and quicker.

For authors who insert index keywords all at once, provides two features that can be quite useful ("Index Entry Mode" and "saved layouts"). These features help authors add index keywords more efficiently.

two features for more efficient indexing
  • Index Entry Mode 's Index Entry Mode lets you add index keywords to the Index window pane more quickly than you could otherwise. With the Index Entry Mode, you do not need to move around the interface as much; you simply click in the topic where you want to insert an index keyword and start typing (without typing over your topic content).
  • Layouts lets you save different layouts of the interface. So you could have one layout saved for when you are creating topics and adding content, and another layout for when you are indexing. In your indexing layout, you probably would want to have the Index window pane open.
 what to do

When you are near the end of the project development process and ready to create your index, do the following.

    • Ribbon In the Reference section select Index Window.
    • Keyboard Shortcut Press F9.

    The Index window pane opens.

  2. In the Window Layout section select Save Layout>Save Layout As.
  3. Give the layout a name such as "Index" and click OK.
  4. Open a topic and in the local toolbar of the XML Editor click .
  5. Insert your index keywords.
  6. When you are finished inserting index keywords, turn off the Index Entry Mode (click again).
  7. In the future, if you need to insert more index keywords, select the Window ribbon. In the Window Layout section select Layout>[Name of Layout].
project—define early

There are tasks that you should perform early in the project development process to ensure that your project is as successful as possible. These tasks include the following.

create a documentation plan

Using a text editor or word processing program, create a document containing as much information about your project as possible. Ask and answer as many questions as you can think of, such as:

  • Who is my audience?
  • Do I have more than one audience, and will I need different outputs for each audience?
  • What subjects will I include?
  • What do I want the final output to look like, and how will it behave (i.e., opens within a software program, opens in its own viewer window, etc.)?
  • How many types of output will I have (i.e., online outputs, printed outputs)?
  • Who will review and test the project?
 develop a style guide

Create a document with style "rules" about your project. By creating a detailed style guide, you can keep your project consistent throughout and avoid repeating work later. Your style guide might include information such as:

  • Spellings of certain terms
  • Abbreviations to be used
  • Formatting specifications (e.g., always use bold font for commands, table headings should have only the first letter capitalized).
  • Remember, the better you define your project ahead of time, the better the outcome will be.

 structure bars

When you are working on a topic in the XML Editor, it is a good idea to make sure the structure bars are turned on. There are two types of structure bars (tag bars and span bars). These bars can help keep your topics less cluttered as you edit them (e.g., you can see condition tags applied to content by looking at the bar rather than having the condition tag markers show up in the content itself). Also, many tasks can be completed much quicker by using these bars. For more information see About Structure Bars.

 styles—preferred over local formatting

When you format your topics to make them look the way you want, you can do so in a couple of different ways.

  • Local Formatting   See Local Formatting.
  • External Stylesheets (Recommended) Using external stylesheets instead of local formatting helps you work faster, more efficiently, and with more consistency in your topics. If you need to make changes to the formatting in the future, you do not need to change the formatting in each topic (as you would with direct formatting). Instead, you only need to change the formatting properties for that style in the appropriate stylesheet. In , you can create and use stylesheets at the topic level, as well as at the table level. See About Styles and Stylesheets.

styles—use master stylesheets

Applying stylesheets at the topic level is attractive because you can associate different stylesheets with different topics, even associating multiple stylesheets with a topic. However, in most cases you will use a single stylesheet for an entire project or target. Therefore, it is usually better to use a master stylesheet at one of those levels. That way, you do not need to manually associate the stylesheet with a topic before using the styles in it.

See Applying Master Stylesheets.

 targets—run batch targets when not logged on

Because scheduled tasks in batch targets use Windows Task Scheduler, the settings in that utility are applied. By default, scheduled tasks will run only if you are logged on. However, you can change this setting in Windows Task Scheduler so that the batch runs whether you are logged on or not. For more information about batch targets, see Building Output Using a Batch Target.

what to do

Complete the following steps if you want to be able to log off and still automatically run scheduled batch targets.

  1. Open Windows Task Scheduler. The steps may be different depending on the operating system. For example, in Windows 7 click the Start button and select All Programs>Accessories>System Tools>Task Scheduler.
  2. Select Task Scheduler Library to see a list of your scheduled tasks.
  3. In the list right-click the appropriate task and click Properties.
  4. Click Run whether user is logged on or not (in Windows 7 this is on the General tab).
  5. Click OK.
  6. You will be prompted to enter your Windows login password.
  7. When you are finished working for the day, log off your computer (instead of shutting down completely). The task will run as scheduled.
 warning topics—view and test as you work

You can save a lot of time developing a project by viewing and testing topics as you work on them, as opposed to waiting until you build the final output. Although there are certain features that require you to build the entire output before testing them (e.g., table of contents, index, cross-references, browse sequences, Help control links), there are other features that you can test for individual topics by simply previewing the topic. This includes clicking drop-down or expanding text hotspots, viewing condition tags (included or excluded from the topic), and getting a sneak preview of how the topic will look in the output. If you take the time to view and test topics as you create them, it could save you time in the end.

To preview and test a topic, simply open the topic and click in the local toolbar of the XML Editor. If you click the face of the button, the topic preview is shown based on the format specified in the primary target. If you click the down arrow, you can select any of the targets in your project from a menu. The topic preview is then displayed using the output format specified in that target. See Previewing Topics and Testing Output.

The topic preview feature is not available for Microsoft Word or Adobe FrameMaker formats.

 topics and table of contents—create simultaneously

You can create new topics in several different ways. With most methods, the topic is created and added to the Content Explorer. Then, if you want the topic to be included in the table of contents (TOC), you need to create an entry in the TOC and link it to the topic. However, you can perform both tasks at the same time from the TOC Editor.

benefits of creating topics from the toc editor

Here are some benefits of creating topics and building a TOC at the same time:

  • Using this method lets you accomplish two tasks at once. It simply saves you time.
  • These two tasks seem to feed on one another. When you first start a new project, creating a TOC early on helps you determine specific topics that you want to include in your project. Conversely, thinking of new topics helps you determine the kind of structure you want for your TOC.
 what to do

Follow the steps in the topic Creating New Topics from the TOC Editor. Use the buttons in the local toolbar of the editor to perform other tasks, such as adding TOC books and moving elements around.

 topics for review—use file tags and file list window pane

If you send topics to reviewers from time to time, there are a couple of features that can help streamline the process. By associating status file tags with topics and using the File List window pane to select topics for review, you might find that the process is much quicker and easier.

Example

Let's say you created a status file tag set so that you can mark your topics with the following statuses, depending on where they are in your development cycle: In Progress, Ready for Review, Ready for Translation, and Complete. (For more information and steps see About File Tagging, Adding File Tag Sets, and Creating File Tags.)

As time goes by, you associate the different statuses with your topics. Some of the topics receive a status of "Ready for Review." (For more information and steps see Associating Tags with Files.)

You want to send topics to your reviewer, but only those topics with the status of "Ready for Review." There are different methods for doing this (e.g., opening the properties for each file to see its status). However, the best and easiest way to accomplish this sort of task is to do the following.

First, open the Review ribbon (or you can use the Review toolbar), unless it is already open.

Next, select View>File List to open the File List window pane (see File List Window Pane). Notice that there are several columns with information about each file, and you can drag the divider to see more of the columns.

The column you are interested in is the one showing the status file tags for each topic. One way to see this column is to drag the column dividers to reduce the size of the other columns. However, there is a better way to accomplish this. You can click in the local toolbar to move the column you want to see higher in the order, or you can hide other columns (see Customizing Column Information).

After doing this, the File List window pane might look something like this:

Now that you can see the status tags next to the name of each file, you can sort the list so that all of the tags with a status of "Ready for Review" are grouped together. To do this, click the column heading that holds the status tags.

Next, click the first file with the "Ready for Review" tag. Hold down your SHIFT key and click the last file in that group.

In the Review toolbar click Send for Review. This opens the Send Files for Review Wizard, which automatically contains the files you selected. Then use the wizard as you normally would to send the topics to one or more individuals for review. (For more information and steps see Sending Topics for Review.)
