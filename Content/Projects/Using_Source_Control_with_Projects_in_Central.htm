Single-bound Projects

Single-bound Projects How to Use Single-Binding
  1. Open the project in MadCap Flare.
  4. Upload (bind) your project to Central. See Uploading (Binding) Projects. See Uploading (Binding) Projects.
  5. In Central, assign users to the new project and make sure they have “Import/Pull” and “Push” permissions. This enables users to import the project using the MadCap Central window pane. It also allows users to push content to the project. See Assigning Users to Teams and Projects and Setting User Permissions. See the online Help. See the Central online Help.
  6. Newly assigned users must now import the project from Central using the MadCap Central window pane in Flare. This allows each user to work on the project locally. See Importing Projects. See Importing Projects from Central.

  7. In Flare, make changes to your files. After you make changes, do the following:

    • Push Upload your changes to Central. To do this, click Push in the Flare Source Control ribbon, or click in the MadCap Central window pane. See Pushing Content. See Pushing Content to Central.
    • Pull Download your teammates' changes to your local project. To do this, click Pull in the Flare Source Control ribbon. See Pulling Content. See Pulling Content from Central.
  8. (Optional) You may encounter file conflicts if multiple users are pushing content to Central. If this happens, pull from Central, then use the Resolve Conflicts dialog to accept or reject changes. See Merging Source Control Files—Git. See the topic for merging Git source control files in the Flare online Help.

  9. When you are done making changes, push your final changes to Central. To do this, click Push in the Flare Source Control ribbon, or by clicking in the MadCap Central window pane. See Pushing Content. See Pushing Content to Central.

    If multiple users are pushing content Central, a designated user should do the final push of the day.

Dual-bound Projects

Dual-bound Projects Note

Just because you have a previous third-party binding to a project, this does not mean you must keep working that way. In other words, you always have the option of unbinding your Flare project from that third-party source control provider and then working in a single-bound model instead. But if you desire to continue working with the other third-party tool, you can do so by using the dual-bound model.

 How to Use Dual-Binding
  1. Open the project in MadCap Flare.
  2. Be sure your project is already bound to a third-party source control provider. See Binding a Project to Source Control. See the Flare online Help.

  5. Upload (bind) your project to Central. See Uploading (Binding) Projects. See Uploading (Binding) Projects.

    Because you are dual-bound, Flare will prompt you to check out your Flare project file before you can upload the project. This allows Central to add a Central ID number to the project file when you upload the project. This ID number links the project to Central.

  6. Check in the Flare project file to your third-party source control provider. This adds the Central ID in the project file to your source control repository, making it available to other users. See Checking In, Committing, and Submitting Source Control Files. See the Flare online Help.
  7. In Central, assign users to the new project and make sure they have “Push” permissions. This gives the users the ability to push content to the project. See Assigning Users to Teams and Projects and Setting User Permissions. See the online Help. See the Central online Help.

  8. Newly assigned users must now do one of the following, depending on whether or not they already have a copy of the Flare project on their local machine:

    • Have a Local Copy Get the latest version of the Flare project file from the third-party source control provider. See Updating or Getting the Latest Version of Source Control Files. See the Flare online Help.

    • Do Not Have a Local Copy Import the project from the third-party source control provider using the Import Project from Source Control Wizard. See Importing a Project from Source Control. See the Flare online Help.

    Either option will give users the most current version of the project file, with the Central ID. This links each user's local copy of the project to Central.

  9. In Flare, make changes to your files. You should manage all of your file changes using your third-party source control provider, using the following actions:

    • Check In Check in your changes to source control. See the Flare online Help. See Checking In, Committing, and Submitting Source Control Files.
    • Get Latest Get the latest version of your teammates' changes from source control and add them to your local project. See the Flare online Help. See Updating or Getting the Latest Version of Source Control Files.

  10. When you are done making changes, push your final changes to Central. To do this, click in the MadCap Central window pane. See Pushing Content. See Pushing Content to Central.

 How to Use Dual-Binding—Git
  1. Open the project in MadCap Flare.
  2. Be sure your project is already bound to Git. See Binding a Project to Git. See the Flare online Help.

  5. Upload (bind) your project to Central. See Uploading (Binding) Projects. See Uploading (Binding) Projects.

    When you bind the project to Central, a Central ID number is added to the project file. This ID number links the project to Central.

  6. Push the Flare project file to Git. This adds the Central ID in the project file to your Git repository, making it available to other users. See Pushing Files to a Remote Repository—Git. See the Flare online Help.
  7. In Central, assign users to the new project and make sure they have “Push” permissions. This gives the users the ability to push content to the project. See Assigning Users to Teams and Projects and Setting User Permissions. See the online Help. See the Central online Help.

  8. Newly assigned users must now do one of the following, depending on whether or not they already have a copy of the Flare project on their local machine:

    • Have a Local Copy Pull the latest version of the Flare project file from Git. See Pulling Files from a Remote Repository—Git. See the Flare online Help.

    • Do Not Have a Local Copy Import the project from Git using the Import Project from Source Control Wizard. See Importing a Project from Git. See the Flare online Help.

    Either option will give users the most current version of the project file, with the Central ID. This links each user's local copy of the project to Central.

  9. In Flare, make changes to your files. When working in a Git/Central dual-bound situation, you can make changes to both Central and Git, using the following actions:

    • Push Push your changes to source control. See the Flare online Help. See Pushing Files to a Remote Repository—Git.

    • Pull Pull your teammates' changes from Git and add them to your local project. See the Flare online Help. See Pulling Files from a Remote Repository—Git.

    It does not matter which location (i.e., Git or Central) you push to or pull from first, as long as you push the files to both Git and Central and that everyone on your team uses the same workflow.

  10. (Optional) If the changes in Git or Central become out-of-sync with each other (i.e., changes are made in one location but not the other), the Resolve Conflicts dialog opens. If this happens, use the dialog to accept or reject other users' changes. See the Flare online Help. See Merging Source Control Files—Git.

  11. When you are done making changes, push your final changes to Central. To do this, click in the MadCap Central window pane. See Pushing Content. See Pushing Content to Central.

There are several different workflows you can use when working with a Git/Central dual-binding. The steps above are only one example of how you might perform this process. Because of the potential for file conflicts, it is essential that you establish a workflow for using a Git/Central dual-binding, and make sure that all your team members follow the same steps.

Examples Important Notes

If your project is dual-bound, you can create new branches in Flare (see Creating Branches—Git see the Flare online Help). However, you can only push files from the Master branch to Central. Single-bound projects do not support branches.

If you switch branches in Flare and try to push changes to Central, but the branch does not have a recognized Central ID associated with it, Flare will prompt you to reload your project. You will need to copy your changes to the master branch and push your changes again.

If you are using a Git/Central dual-binding, and you encounter conflicts in one repository (i.e., Git or Central), you will likely encounter them in the other location as well. Conflicts in one repository will most likely need to be resolved in the other repository. See Merging Source Control Files—Git. See the Flare online Help.

For more information about managing dual-bound projects using Flare's built in source control system, see the Flare online Help.
